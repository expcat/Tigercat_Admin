# 用户 / 角色 / 设置核心流程 E2E 回归计划

最后更新：2026-05-28

---

## 目标

为 Tigercat Admin 补齐一份可以直接落地的端到端回归计划，优先覆盖以下两类风险：

- 双端实现虽然共用后端，但页面交互、控件接线和权限守卫仍可能出现 React / Vue 偏差。
- 用户、角色、设置属于后台最高频工作流，一旦回归失效，会直接影响后续通知、任务、审计等页面的可用性。

本计划当前只定义回归范围、执行顺序、数据约束和通过标准，不在本轮任务中直接引入 Playwright、Cypress 或其他 E2E 运行框架。

当前已落地的第一批基座：

- 根目录已引入 Playwright 工作区配置，可通过 `pnpm e2e`、`pnpm e2e:react`、`pnpm e2e:vue` 运行。
- 已补共享认证夹具，覆盖 React / Vue 双端的登录、受保护路由重定向、会话恢复与用户页访问烟测。
- E2E 运行时 API 默认切到 `Infrastructure:UseInMemory=true` 与 `Database:Provider=InMemory`，避免依赖本机 Redis。

## 范围

首轮回归范围限定为三条业务主线：

1. 认证与会话基线：登录、登出、受保护路由跳转、会话失效后的回退。
2. 用户与角色主流程：用户列表、创建、编辑、状态切换、角色分配、角色权限配置。
3. 系统设置主流程：设置读取、批量保存、恢复默认值、关键主题与安全字段的持久化回显。

暂不纳入首轮 E2E 的内容：

- 文件真实上传与对象存储。
- 审计日志、通知中心、任务面板等扩展页面。
- 导出文件内容的字节级校验。

这些能力后续应在主流程稳定后增量加入，不和首轮核心回归绑在一起。

## 执行矩阵

同一套业务断言要覆盖两套前端实现：

| 维度     | React                                     | Vue                                       |
| -------- | ----------------------------------------- | ----------------------------------------- |
| 前端工程 | `Tigercat.Admin.React`                    | `Tigercat.Admin.Vue`                      |
| 路由基线 | `/login`、`/users`、`/roles`、`/settings` | `/login`、`/users`、`/roles`、`/settings` |
| 最小目标 | 主流程可执行，结果与 API 状态一致         | 主流程可执行，结果与 API 状态一致         |

数据库基线分两档：

- 本地开发与手工回归：默认使用 SQLite，保持与日常开发一致。
- 自动化 E2E 运行：优先使用 SQLite；InMemory 仅用于极轻量烟测，不作为主回归基线。

PostgreSQL 不作为首轮 E2E 的默认执行环境，但在后续部署前应增加独立的生产冒烟链路。

## 环境前提

执行回归前应满足以下约束：

- API 已启动，且数据库 provider 明确为 `Sqlite` 或测试指定值。
- 默认种子管理员可用：`admin / admin123`。
- 前端使用待验证实现单独启动，避免 React 与 Vue 在同一轮里混淆结果。
- Redis 可用，避免登录、权限和审计链路因为基础设施缺失而出现伪失败。

推荐启动方式：

```bash
# 方式 1：统一通过 Aspire
cd Tigercat.Aspire
dotnet run

# 方式 2：单独启动 API + 指定前端
cd Tigercat.Admin.Api
dotnet run

cd Tigercat.Admin.React
pnpm dev

# 或
cd Tigercat.Admin.Vue
pnpm dev
```

## 数据约束

为了让 E2E 可重复执行，测试数据需要遵守固定约束：

- 所有自动化创建的数据统一使用 `e2e_` 前缀，例如 `e2e_user_ops`、`e2e_role_ops`。
- 用户、角色测试应尽量自清理；若清理失败，也要保证下一次运行仍可用唯一后缀规避冲突。
- 设置测试只改动可逆字段，例如 `site.name`、`auth.sessionTimeout`、`theme.mode`、`theme.primaryColor`、`theme.compactMode`。
- 涉及权限收缩的用例不得修改默认 `Admin` 角色，避免污染后续回归基线。

## 首轮用例清单

### 认证与会话

1. 未登录访问受保护路由时，自动跳转到 `/login`。
2. 使用 `admin / admin123` 登录成功后进入 `/dashboard`。
3. 登录后刷新页面，会话仍能恢复，导航与权限菜单正常展示。
4. 执行退出登录后返回 `/login`，再次访问 `/users` 会被拦截。

### 用户流程

1. 打开用户页并加载分页列表，默认管理员记录可见。
2. 创建带角色的新用户，提交后列表出现新记录。
3. 编辑用户显示名或状态，列表与详情弹层回显一致。
4. 为用户调整角色后，重新打开编辑弹层可看到变更已持久化。
5. 执行单条删除或批量删除后，列表不再出现目标用户。

### 角色流程

1. 打开角色页并加载列表，`Admin`、`Editor`、`Viewer` 可见。
2. 创建新角色并勾选权限树，保存后列表出现新角色。
3. 编辑角色名称或描述，重新打开弹层可看到更新结果。
4. 修改角色权限后，角色详情与权限摘要保持一致。
5. 删除非系统保留角色后，列表同步更新。

### 设置流程

1. 打开设置页并成功读取分组配置。
2. 修改 `site.name`、`theme.mode`、`theme.primaryColor`、`auth.sessionTimeout` 后保存，刷新页面仍能回显。
3. 触发保存确认弹窗时，取消不会落库，确认后才生效。
4. 对单项设置执行恢复默认值后，界面值与接口返回的 `defaultValue` 一致。
5. 重新登录后再次进入设置页，能看到上一步保存后的持久化结果。

## 推荐落地顺序

为了降低首轮 E2E 建设成本，建议按以下顺序推进：

1. 先做共享登录夹具与会话恢复断言，打通 React / Vue 两端公共前置步骤。
2. 再补用户页 CRUD 主线，因为它最能覆盖表格、弹窗、权限和 API 写操作。
3. 之后补角色权限配置，验证 Tree 与角色分配链路。
4. 最后补设置页的保存、确认和恢复默认值，作为状态持久化闭环。

如果后续引入 Playwright，建议使用 workspace 级项目配置，把 React 与 Vue 作为两个 project，共享同一套 fixture 与断言语义，只在选择器和启动命令层分开。

## 完成标准

该计划进入“已落地”状态的最低标准如下：

1. React 与 Vue 至少各自具备一条可运行的登录 + 用户 CRUD 主流程用例。
2. 用户、角色、设置三条主线都具备成功路径断言，且不依赖手工改数据库。
3. 回归执行结果能明确区分前端交互问题、权限问题和后端接口失败。
4. 新增或修改核心流程时，优先补 E2E 计划或用例，再扩展外围页面。
